1xx информационный ответ
Информационный ответ указывает на то, что запрос получен, понят и находится в процессе обработки. Он предупреждает клиента о необходимости дождаться окончательного ответа. Сообщение не содержит тела. Поскольку стандарт HTTP/1.0 не определяет никаких кодов состояния 1xx, серверы не должны отправлять ответ 1xx клиенту, совместимому с HTTP/1.0, за исключением экспериментальных условий.

100 Продолжить
Сервер получил заголовки запроса, и клиент должен перейти к отправке тела запроса (в случае запроса, для которого необходимо отправить тело, например, POST- запрос). Отправка большого тела запроса на сервер после того, как запрос был отклонен из-за неподходящих заголовков, была бы неэффективной. Чтобы сервер проверил заголовки запроса, клиент должен отправить Expect: 100-continueзаголовок в своем первоначальном запросе и получить 100 Continueкод состояния в ответе, прежде чем отправлять тело. Если клиент получает код ошибки, например, 403 (Forbidden) или 405 (Method Not Allowed), то он не должен отправлять тело запроса. Ответ 417 Expectation Failedуказывает на то, что запрос следует повторить без Expectзаголовка, поскольку он указывает на то, что сервер не поддерживает ожидаемые значения (это, например, случай серверов HTTP/1.0). [ 2 ] : §10.1.1 
101 Протоколы коммутации
Заявитель обратился к серверу с просьбой сменить протокол, и сервер согласился это сделать.
102 Обработка ( WebDAV ; RFC 2518)
Запрос WebDAV может содержать множество подзапросов, включающих операции с файлами, что требует длительного времени для выполнения запроса. Этот код указывает на то, что сервер получил и обрабатывает запрос, но ответа пока нет. [ 3 ] Это предотвращает тайм-аут клиента и предположение о потере запроса. Код состояния устарел. [ 4 ]
103 Первые подсказки (RFC 8297)
Используется для возврата некоторых заголовков ответа перед окончательным HTTP-сообщением. [ 5 ]
2xx успеха
Статус "успешно" означает, что действие, запрошенное клиентом, было получено, понято и принято. [ 1 ]

200 ОК
Стандартный ответ для успешных HTTP-запросов. Фактический ответ будет зависеть от используемого метода запроса . В случае запроса GET ответ будет содержать сущность, соответствующую запрошенному ресурсу. В случае запроса POST ответ будет содержать сущность, описывающую или содержащую результат действия.
201 Создано
Запрос выполнен, в результате чего создан новый ресурс. [ 6 ]
202 Принято
Запрос принят к обработке, но обработка еще не завершена. Запрос может быть рассмотрен, а может и не быть рассмотрен, и может быть отклонен после завершения обработки.
203 Неавторитетная информация (начиная с HTTP/1.1)
Сервер представляет собой преобразующий прокси (например, веб-акселератор ), который получил 200 OK от своего источника, но возвращает модифицированную версию ответа источника. [ 2 ] : §15.3.4  [ 2 ] : §7.7 
204 Нет контента
Сервер успешно обработал запрос и не возвращает никакого контента.
205 Сбросить содержимое
Сервер успешно обработал запрос, запросил у отправителя сброс представления документа и не возвращает никакого содержимого.
206 Частичное содержание
Сервер передает только часть ресурса ( последовательность байтов ) из-за заголовка диапазона, отправленного клиентом. Заголовок диапазона используется HTTP-клиентами для возобновления прерванных загрузок или разделения загрузки на несколько одновременных потоков.
207 Многостатусный (WebDAV; RFC 4918)
Тело сообщения, которое следует далее, по умолчанию представляет собой XML- сообщение и может содержать несколько отдельных кодов ответа в зависимости от количества сделанных подзапросов. [ 7 ]
208 уже зарегистрированных случаев (WebDAV; RFC 5842)
Участники связывания DAV уже были перечислены в предыдущей части ответа (многостатусного) и не будут включены повторно.
Использован 226-й внутренний модуль (RFC 3229)
Сервер выполнил запрос на ресурс, и ответ представляет собой результат одной или нескольких манипуляций с экземпляром, примененных к текущему экземпляру. [ 8 ]
перенаправление 3xx
Статус 3xx указывает на то, что клиенту необходимо выполнить дополнительные действия, как правило, перенаправление URL-адреса , для завершения запроса. [ 1 ] Пользовательский агент может выполнить дополнительное действие без взаимодействия с пользователем, если в дополнительном запросе используется метод GET или HEAD. Пользовательский агент должен предотвращать циклические перенаправления. [ 2 ] : §15.4 

300 вариантов ответа
Указывает на несколько вариантов ресурса, из которых клиент может выбрать (посредством согласования содержимого, управляемого агентом ). Например, этот код можно использовать для отображения нескольких вариантов видеоформатов, для перечисления файлов с различными расширениями или для предложения разрешения неоднозначности значений слов .
301 Перемещено навсегда
Целевой адрес ссылки был перемещен таким образом, чтобы запрос и будущие аналогичные запросы перенаправлялись на указанный URI . Если клиент обладает возможностью редактирования ссылок, он должен обновить ссылки на URL-адрес запроса. Ответ кэшируется, если не указано иное. За исключением запроса GET, тело запроса должно содержать гиперссылку на новый(е) URL(ы). За исключением запроса GET или HEAD, клиент должен запросить у пользователя подтверждение перед перенаправлением. [ 9 ]
Этот код считается лучшей практикой для перехода пользователей с HTTP на HTTPS . И Bing , и Google рекомендуют использовать этот код для изменения URL страницы в результатах поиска, при условии, что этот URL будет изменен навсегда и не планируется его изменение в ближайшее время. [ 10 ] [ 11 ]
Найдено 302
Указывает, что ресурс доступен через альтернативный URL, указанный в поле заголовка Location . Спецификация HTTP/1.0 (в которой использовалась фраза причины "Moved Temporarily") требовала от клиента перенаправления тем же методом [ 12 ] , но популярные браузеры вместо этого изменили запрос на GET [ 13 ] . По этой причине HTTP/1.1 ( RFC  2616 ) добавил два кода состояния: 303, который требует изменения запроса на GET, и 307, который сохраняет исходный тип запроса. Несмотря на большую ясность, обеспечиваемую этим уточнением, код 302 по-прежнему используется в веб-фреймворках для сохранения совместимости с браузерами, которые не поддерживают HTTP/1.1 [ 14 ] [ 2 ] : §15.4  В результате RFC  7231 (обновление RFC  2616 ) изменяет определение, чтобы позволить пользовательским агентам переписывать POST на GET [ 15 ] .
303 См. Другое (начиная с HTTP/1.1)
Если сервер отвечает на POST-запрос или другой неидемпотентный запрос этим кодом и полем заголовка Location , ожидается, что клиент отправит GET-запрос к указанному местоположению. Чтобы инициировать запрос к целевому ресурсу тем же способом, сервер отвечает кодом 307.
Использование этого кода было предложено [ 16 ] как один из способов ответа на запрос URI , который идентифицирует реальный объект в соответствии с теорией семантической сети (другой способ — использование хэш-URI [ 17 ] ). Например, если http://www.example.com/id/aliceидентифицирует человека, Алису, то для сервера было бы неуместно отвечать на GET-запрос кодом 200 OK, поскольку сервер не мог бы предоставить саму Алису. Вместо этого сервер ответил бы кодом 303, чтобы перенаправить на URI, который предоставляет описание человека по имени Алиса.
Иногда этот код используется при предоставлении веб-API на основе HTTP , которому необходимо немедленно ответить вызывающей стороне, но при этом продолжить выполнение асинхронно, например, при длительном преобразовании изображения. Веб-API предоставляет URI проверки состояния, который позволяет клиенту проверить статус операции. После завершения ответ может содержать этот код состояния и URI перенаправления к конечному результату. [ 18 ]
304 Не изменено
Указывает на то, что ресурс не был изменен с момента версии, указанной в заголовках запроса If-Modified-Since или If-None-Match. В таком случае нет необходимости повторно передавать ресурс, поскольку у клиента по-прежнему есть ранее загруженная копия.
305 Использовать прокси (начиная с HTTP/1.1)
Запрошенный ресурс доступен только через прокси, адрес которого указан в ответе. Из соображений безопасности многие HTTP-клиенты (например, Mozilla Firefox и Internet Explorer ) не соблюдают этот код состояния. [ 19 ]
306 Switch Proxy
Больше не используется. Первоначально означало: «Последующие запросы должны использовать указанный прокси-сервер».
307 Временное перенаправление (начиная с HTTP/1.1)
В этом случае запрос следует повторить с другим URI; однако в будущих запросах следует по-прежнему использовать исходный URI. В отличие от того, как исторически реализовывался код 302, метод запроса не допускается при повторной отправке исходного запроса. Например, POST-запрос следует повторить, используя другой POST-запрос.
308 Постоянное перенаправление
Этот и все будущие запросы должны быть направлены на указанный URI . Код 308 аналогичен поведению кода 301, но не позволяет изменить HTTP-метод . Таким образом, например, отправка формы на ресурс, на который постоянно перенаправляется, может продолжаться без проблем.

Ошибка 404 на Викимедиа
Код состояния 4xx используется в ситуациях, когда ошибка, по всей видимости, вызвана действиями клиента. За исключением случаев ответа на HEAD-запрос, сервер должен включать в ответ сущность, содержащую объяснение ситуации с ошибкой и указание, является ли она временной или постоянной. Эти коды состояния применимы к любому методу запроса . Пользовательские агенты должны отображать пользователю любую включенную сущность.

400 Неверный запрос
Сервер не может или не хочет обрабатывать запрос из-за очевидной ошибки клиента (например, некорректный синтаксис запроса, слишком большой размер, недопустимая структура сообщения запроса или обманчивая маршрутизация запроса).
401 Несанкционированный доступ
Аналогично ошибке 403 Forbidden, но предназначена для случаев, когда требуется аутентификация, но она не удалась или еще не была предоставлена. Ответ должен содержать поле заголовка WWW-Authenticate, содержащее запрос, применимый к запрашиваемому ресурсу. См. базовую аутентификацию доступа и дайджест-аутентификацию доступа . Ошибка 401 семантически означает «неаутентифицирован», у пользователя нет действительных учетных данных для доступа к целевому ресурсу.
402 Требуется оплата
Зарезервировано для будущего использования. Изначально предполагалось, что этот код может использоваться как часть какой-либо формы цифровых денег или схемы микроплатежей , как это предлагалось, например, GNU Taler [ 20 ] , но этого пока не произошло, и этот код не получил широкого распространения. Google Developers API использует этот статус, если конкретный разработчик превысил дневной лимит запросов [ 21 ] . Sipgate использует этот код, если на счету недостаточно средств для начала вызова [ 22 ] . Shopify использует этот код, когда магазин не оплатил комиссию и временно заблокирован [ 23 ] . Stripe использует этот код для неудачных платежей, когда параметры были правильными, например, заблокированных мошеннических платежей [ 24 ] .
403 Запрещено
Запрос был действительным, но сервер отклонил действие. Это может быть связано с тем, что у пользователя нет разрешения на доступ к ресурсу, ему необходима учетная запись или он пытается совершить запрещенное действие (например, создать дубликат записи там, где разрешена только одна). Этот код также обычно используется, если запрос обеспечил аутентификацию путем ответа на запрос в поле заголовка WWW-Authenticate, но сервер не принял эту аутентификацию. Запрос не следует повторять.
Этот код отличается от 401 тем, что, в то время как 401 возвращается, когда клиент не прошел аутентификацию и подразумевает, что после действительной аутентификации может быть возвращен успешный ответ, 403 возвращается, когда клиенту не разрешен доступ к ресурсу, несмотря на предоставление аутентификации, например, из-за недостаточных прав аутентифицированной учетной записи.
Веб -сервер Apache возвращает 403 в ответ на запрос URL-адресов [ 25 ] , соответствующих каталогу файловой системы , когда отображение списка каталогов отключено и отсутствует директива Directory Index , указывающая на существующий файл, который должен быть возвращен браузеру. Некоторые администраторы настраивают расширение Mod proxy для блокировки таких запросов, и это также возвращает 403. IIS реагирует аналогичным образом, когда отображение списка каталогов запрещено на этом сервере. В WebDAV 403 возвращается, если клиент отправил запрос PROPFIND, но не отправил также необходимый заголовок Depth или отправил заголовок Depth со значением бесконечности. [ 25 ]
Эта ошибка может появиться по следующим причинам:
Недостаточные права доступа: Наиболее распространенная причина заключается в том, что у пользователя отсутствуют необходимые права доступа к ресурсу.
Требуется аутентификация: В некоторых случаях сервер требует аутентификации для доступа к определенным ресурсам.
Ограничения по IP-адресам: Сервер также может ограничивать доступ к определенным IP-адресам или диапазонам IP-адресов.
Конфигурация сервера: В настройках сервера можно запретить доступ к определенным файлам, каталогам или разделам веб-сайта. Это может быть вызвано неправильной конфигурацией или преднамеренными ограничениями, установленными администратором сервера.
Блокировка брандмауэром или программным обеспечением безопасности: Этот код может появиться, если брандмауэр или программное обеспечение безопасности блокирует доступ к ресурсу. Это может произойти из-за политик безопасности, обнаружения вредоносных программ или других мер безопасности.
Ограничение скорости или слишком большое количество запросов: Когда клиент отправляет чрезмерное количество запросов за короткий промежуток времени, сервер может ответить кодом 403, чтобы предотвратить злоупотребления или атаки типа «отказ в обслуживании».
404 Не найдено
Основная статья: HTTP 404
Запрошенный ресурс не найден, но может быть доступен в будущем. Повторные запросы от клиента допустимы.
Метод 405 запрещен
Для запрашиваемого ресурса не поддерживается определенный метод запроса (например, GET-запрос к форме, требующей отправки данных методом POST , или PUT-запрос к ресурсу только для чтения).
406 Неприемлемо
Запрашиваемый ресурс способен генерировать только контент, неприемлемый согласно заголовкам Accept, отправленным в запросе. См. Согласование контента .
407 Требуется аутентификация прокси-сервера
Клиент должен сначала пройти аутентификацию у прокси-сервера .
408 Время ожидания запроса истекло
Сервер превысил время ожидания запроса. Согласно спецификации HTTP: «Клиент не отправил запрос в течение времени, которое сервер был готов ожидать. Клиент МОЖЕТ повторить запрос без изменений в любое последующее время».
409 Конфликт
Указывает на то, что запрос не может быть обработан из-за конфликта в текущем состоянии ресурса, например, конфликта редактирования между несколькими одновременными обновлениями. [ 26 ]
410 исчезло
Указывает на то, что запрошенный ресурс ранее использовался, но больше недоступен и больше не будет доступен. Этот код следует использовать, когда ресурс был преднамеренно удален и его необходимо очистить. При получении кода состояния 410 клиент не должен запрашивать ресурс в будущем. Клиенты, такие как поисковые системы, должны удалить ресурс из своих индексов. В большинстве случаев клиентам и поисковым системам не требуется очищать ресурс, и вместо этого можно использовать код «404 Not Found».
411 Требуемая длина
В запросе не указана длина его содержимого, которая необходима для запрашиваемого ресурса.
412 Предварительное условие не выполнено
Сервер не соответствует одному из предварительных условий, указанных отправителем в полях заголовка запроса.
413 Слишком большой размер контента
Запрос превышает возможности или готовность сервера его обработать. Ранее назывался «Слишком большой размер сущности запроса» и «Слишком большой размер полезной нагрузки». [ 27 ] : §10.4.14  [ 2 ] : §15.5.14 
414 URI Too Long
Предоставленный URI оказался слишком длинным для обработки сервером. Часто это происходит из-за того, что слишком много данных кодируется в качестве строки запроса GET-запроса, в этом случае его следует преобразовать в POST-запрос. Ранее это называлось "Слишком длинный URI запроса". [ 27 ] : §10.4.15 
415 Неподдерживаемый тип носителя
В запросе указывается тип носителя , который сервер или ресурс не поддерживают. Например, клиент загружает изображение в формате image/svg+xml , но сервер требует, чтобы изображения использовали другой формат.
Диапазон 416 неудовлетворительный
Клиент запросил часть файла ( побайтовое предоставление ), но сервер не может предоставить эту часть. Например, если клиент запросил часть файла, которая выходит за пределы файла. Ранее это называлось "Запрошенный диапазон не удовлетворяется". [ 27 ] : §10.4.17 
417 Ожидание не оправдалось
Сервер не может соответствовать требованиям поля Expect в заголовке запроса. [ 28 ]
418 Я — чайник (RFC 2324, RFC 7168)
Этот код был определен в 1998 году как одна из традиционных первоапрельских шуток IETF в RFC 2324, Hyper Text Coffee Pot Control Protocol , и не предполагается, что он будет реализован реальными HTTP-серверами. RFC указывает, что этот код должен возвращаться чайниками, запрошенными для приготовления кофе. [ 29 ] Этот HTTP-статус используется в качестве пасхального яйца на некоторых веб-сайтах, например, пасхальное яйцо «Я чайник» на Google.com . [ 30 ] [ 31 ] Иногда этот код состояния также используется в качестве ответа на заблокированный запрос, вместо более подходящего 403 Forbidden. [ 32 ] [ 33 ]

424 Неудачная зависимость (WebDAV; RFC 4918)
Запрос не удался, потому что он зависел от другого запроса, и этот запрос не удался (например, PROPPATCH). [ 7 ]

Это указывает на то, что сервер не желает рисковать обработкой запроса, который может быть повторен.

Исходный сервер требует, чтобы запрос был условным. Это предназначено для предотвращения проблемы «потерянного обновления», когда клиент получает состояние ресурса, изменяет его и отправляет обратно на сервер, в то время как третья сторона уже изменила состояние на сервере, что приводит к конфликту. [ 34 ]

ошибка сервера 5xx
Статус 5xx указывает на то, что сервер осведомлен о возникшей ошибке или не в состоянии выполнить запрос. За исключением случаев ответа на HEAD-запрос, сервер должен включить в ответ сущность, содержащую объяснение ситуации с ошибкой, и указать, является ли она временной или постоянной. Аналогично, пользовательские агенты должны отображать пользователю любую включенную сущность. Эти коды ответа применимы к любому методу запроса .

Ошибка 500 (внутренняя ошибка сервера)
Общее сообщение об ошибке, выдаваемое в случае возникновения непредвиденной ситуации, когда более конкретное сообщение не подходит.
501 Не реализовано

502 Неверный шлюз

Сервис 503 недоступен
Сервер не может обработать запрос (потому что он перегружен или отключен для технического обслуживания). Как правило, это временное состояние. [ 37 ]
504 Таймаут шлюза
Сервер выступал в качестве шлюза или прокси-сервера и не получил своевременного ответа от вышестоящего сервера.
Ошибка 505: Версия HTTP не поддерживается
Сервер не поддерживает версию HTTP, использованную в запросе.

Прозрачное согласование содержимого запроса приводит к циклической ссылке . [ 38 ]
507 Недостаточно места для хранения данных (WebDAV; RFC 4918)

Клиенту необходимо пройти аутентификацию для получения доступа к сети. Предназначен для использования путем перехвата прокси-серверов, используемых для управления доступом к сети (например, « порталы авторизации », используемые для требования согласия с условиями обслуживания перед предоставлением полного доступа в Интернет через точку доступа Wi-Fi ).
